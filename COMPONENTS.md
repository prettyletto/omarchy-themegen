# COMPONENTS

## Theme Surface Policy

This document lists supported Omarchy Theme Surfaces. It does not define plugins, interfaces, or exporters.

A Theme Surface is supported when it is consumed by local Omarchy behavior, generated from installed Omarchy templates, or required by product scope.

VS Code is excluded even though Omarchy has a sync script and many themes include `vscode.json`.

## Surface Groups

Component-mix mode presents groups first, then allows per-surface overrides.

Groups are UX structure only. They must resolve into one Theme Model before export.

## Required Direct Theme Files

### `colors.toml`

Canonical color contract for Omarchy template generation.

Keys:

- `accent`
- `cursor`
- `foreground`
- `background`
- `selection_foreground`
- `selection_background`
- `color0` through `color15`

### Backgrounds

The exported theme must include the source image under `backgrounds/`.

The source image should be preserved when possible. Preview assets may be cropped or resized separately.

Local Omarchy background cycling reads current theme backgrounds from `~/.config/omarchy/current/theme/backgrounds/`.

### Preview Assets

Generated files:

- `preview.png` at 1800x1012;
- `preview-unlock.png` at 1920x1080;
- `unlock.png` as a valid Plymouth/SDDM logo image.

Known consumers:

- theme menu uses `preview.png`, then `preview.jpg`, then first background image;
- unlock menu uses `preview-unlock.png`;
- `omarchy-plymouth-set-by-theme` uses `unlock.png` with `colors.toml`.

### Neovim

Generated file: `neovim.lua`.

Local Neovim consumes it through `~/.config/nvim/lua/plugins/theme.lua`, symlinked to `~/.config/omarchy/current/theme/neovim.lua`.

Generation must detect the installed Aether/AetherNvim palette contract and target that contract only. If detection fails, Neovim generation fails clearly.

### README

Generated product artifact. Omarchy does not require it.

## Optional/Fixed Direct Files

### `icons.theme`

Fixed/default value only. Do not derive it from palette colors.

If omitted, local Omarchy falls back to `Yaru-blue`.

### `light.mode`

Written only for explicit light-theme exports.

Local Omarchy uses this marker in `omarchy-theme-set-gnome` to select GNOME `prefer-light` and `Adwaita`.

## Template-Generated Surfaces

These are generated by installed Omarchy templates when `colors.toml` exists and the file is not already present.

### Waybar

Installed template maps foreground and background.

No layout changes.

### Ghostty

Installed template maps background, foreground, cursor, selection colors, and terminal palette.

No terminal behavior settings.

### Alacritty / Foot / Kitty

Installed templates map terminal colors.

No terminal behavior settings.

### Hyprland

Installed template maps accent to active window border and active group border.

No keybindings, monitors, animations, window rules, or behavior.

### Hyprlock

Installed template maps background, inner color, outer color, font color, and check/accent color.

No auth behavior or layout redesign.

### Mako

Installed template maps text color, border color, and background color.

No notification behavior settings.

### btop

Installed template maps background, foreground, accent, and terminal-palette-derived graph colors.

No layout or metric settings.

### Walker / SwayOSD / Chromium / Helix / gum / Obsidian / Hyprland Preview Share Picker

Generated from installed Omarchy templates. Treat mappings as Omarchy-owned.

### Keyboard RGB

Installed `keyboard.rgb.tpl` uses `{{ accent }}`.

Omarchy applies it only when supported keyboard tooling/hardware exists. It must not influence palette decisions.

## Excluded Surfaces

Exclude even if seen in community themes:

- VS Code;
- Vencord/Discord;
- Spotify/Spicetify;
- Steam;
- Firefox/Zen;
- Wofi;
- Warp;
- Zed/Aether custom files;
- GTK/QT files beyond Omarchy's GNOME behavior;
- Cava;
- Hyprbars;
- shell config files;
- arbitrary user scripts;
- repository metadata.

## Implementation Warnings

- Do not create a Theme Surface plugin system.
- Do not create one exporter per surface.
- Do not let surface variants bypass Theme Model validation.
- Do not let palette generation depend on app file formats.
- Do not hand-author files Omarchy templates can generate.
