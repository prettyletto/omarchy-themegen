# Sprint 1 Tasks

## Sprint Goal

Prove the core non-UI contract: `omarchy-themegen` can validate one source image, build a static Theme Model, structurally validate it, and export a self-contained Omarchy Theme Directory without applying it.

This sprint intentionally avoids palette extraction, TUI workflows, browser preview, and component-mix UI. Those depend on a proven Theme Model and exporter.

## Task 1: Bootstrap the `omarchy-themegen` CLI

Type: AFK

Blocked by: None

### What to build

Create the initial Go project and a runnable `omarchy-themegen` binary. The binary should support a minimal command shape that future tasks can extend:

- `omarchy-themegen <image>` defaults to interactive/TUI intent, but may return a clear "TUI not implemented yet" message during this sprint.
- explicit non-interactive export path is available for sprint validation.
- plain-text output is default.
- JSON output is available behind an explicit flag.

Do not implement TUI screens, palette extraction, browser preview, or Omarchy apply.

### Acceptance criteria

- [ ] `go test ./...` passes.
- [ ] `go install` produces an `omarchy-themegen` binary.
- [ ] Running without required arguments prints useful usage text.
- [ ] JSON output is emitted only when explicitly requested.
- [ ] No command invokes `omarchy theme set`.

## Task 2: Validate the Theme Source Image

Type: AFK

Blocked by: Task 1

### What to build

Implement validation for exactly one Theme Source Image per run.

Validation must reject:

- unreadable files;
- non-image files;
- transparent images;
- animated images/video;
- images smaller than 800x450;
- more than one source image.

Validation should warn, not reject, when the image appears UI-heavy.

Image processing may use `magick`; if an operation requires it and it is missing, fail with a clear dependency error.

### Acceptance criteria

- [ ] Opaque still images at or above 800x450 pass validation.
- [ ] Transparent images fail validation.
- [ ] Tiny images fail validation.
- [ ] Non-image inputs fail validation.
- [ ] Multiple image arguments fail validation.
- [ ] UI-heavy detection can produce a warning without blocking export.
- [ ] Missing `magick` is reported clearly for paths that require it.

## Task 3: Represent and Validate the Omarchy Color Contract

Type: AFK

Blocked by: Task 1

### What to build

Implement the minimal Theme Model needed to export a static theme. This task does not generate palettes from the image yet.

The model must represent the required `colors.toml` keys:

- `accent`
- `cursor`
- `foreground`
- `background`
- `selection_foreground`
- `selection_background`
- `color0` through `color15`

All colors must validate as `#RRGGBB`.

Use a static deterministic color set for now so the exporter can be proven before palette extraction exists.

### Acceptance criteria

- [ ] A static Theme Model can be created from a valid source image and explicit theme name.
- [ ] Missing required color keys are rejected.
- [ ] Invalid color formats are rejected.
- [ ] Valid colors serialize to a `colors.toml` content string.
- [ ] The Theme Model is independent of UI state and filesystem output paths.

## Task 4: Export a Self-Contained Omarchy Theme Directory

Type: AFK

Blocked by: Task 2, Task 3

### What to build

Export one validated Theme Model to an Omarchy Theme Directory.

The exported directory must include:

- `colors.toml`;
- `backgrounds/<source-image>`;
- `preview.png`;
- `preview-unlock.png`;
- `unlock.png`;
- `neovim.lua`;
- `README.md`;
- optional fixed/default `icons.theme` if the implementation chooses to emit it.

This sprint may use simple generated placeholder preview/unlock assets as long as they are valid images with the documented dimensions for `preview.png` and `preview-unlock.png`.

Do not generate files that installed Omarchy templates can generate from `colors.toml`.

### Acceptance criteria

- [ ] Export creates a directory usable as `~/.config/omarchy/themes/<normalized-theme-name>`.
- [ ] The source image is copied under `backgrounds/`.
- [ ] `preview.png` is generated at 1800x1012.
- [ ] `preview-unlock.png` is generated at 1920x1080.
- [ ] `unlock.png` is a valid image.
- [ ] `neovim.lua` exists, or the command fails clearly if Aether contract detection is not implemented yet.
- [ ] `README.md` states the theme was generated by `omarchy-themegen` and includes basic install/CLI usage.
- [ ] Export does not call `omarchy theme set`.

## Task 5: Add Structural Validation Before and After Export

Type: AFK

Blocked by: Task 4

### What to build

Implement structural validation for export. Validation must be non-mutating.

Before writing, validate:

- source image passed validation;
- user explicitly provided a theme name;
- normalized theme name is non-empty;
- exactly one Theme Model exists;
- required color keys exist and are valid;
- required assets can be generated.

After writing, validate:

- target directory exists;
- required files exist;
- `colors.toml` contains all required keys;
- generated images are valid files;
- optional `omarchy theme list` discovery check runs only when Omarchy is installed and must not apply the theme.

### Acceptance criteria

- [ ] Validation failures prevent export.
- [ ] Post-export validation catches missing files.
- [ ] Validation never invokes `omarchy theme set`.
- [ ] Validation reports reduced confidence when Omarchy is unavailable.
- [ ] Errors are actionable and identify the failing check.

## Task 6: Implement Safe Overwrite and Backup Behavior

Type: AFK

Blocked by: Task 5

### What to build

Implement overwrite protection for local theme directory export.

Default behavior must refuse to overwrite an existing target. Confirmed replacement must create a timestamped backup before writing the replacement.

`--yes` may confirm overwrite only when backup creation is included. `--yes` must not imply apply.

### Acceptance criteria

- [ ] Export to a new theme name succeeds.
- [ ] Export to an existing theme name fails by default.
- [ ] Confirmed replacement creates a timestamped backup.
- [ ] Replacement does not start unless backup succeeds.
- [ ] Backup path is reported to the user.
- [ ] `--yes` confirms overwrite-with-backup only.
- [ ] No overwrite path invokes Omarchy apply.

## Task 7: Export Finished-Theme Archives

Type: AFK

Blocked by: Task 5

### What to build

Add explicit finished-theme archive export.

The archive must contain the same validated Omarchy Theme Directory content and use the theme directory itself as the archive root. It must not include recipe files, source generation details beyond the normal theme content, license files, `.gitignore`, git metadata, or publishing metadata.

### Acceptance criteria

- [ ] Archive export is explicit.
- [ ] Archive root is `<normalized-theme-name>/`.
- [ ] Archive contains the same required files as the exported theme directory.
- [ ] Archive export can run without writing to `~/.config/omarchy/themes`.
- [ ] Archive export does not include repo/legal metadata by default.
- [ ] Archive export does not invoke Omarchy apply.

## Task 8: Document Sprint-1 CLI Behavior in the Generated README

Type: AFK

Blocked by: Task 4

### What to build

Generate a README for exported themes that reflects the current sprint's supported behavior without promising unimplemented features.

The README must include:

- generated-by notice;
- `go install` installation guidance;
- basic CLI usage for generating/exporting;
- how to place the theme under `~/.config/omarchy/themes/<name>`;
- how to apply the theme with Omarchy manually;
- note that apply is separate from export.

Generation details must be omitted unless explicitly requested by a later recipe/reproducibility task.

### Acceptance criteria

- [ ] README is generated for every exported theme.
- [ ] README does not claim Omarchy requires the README.
- [ ] README does not include source image bytes or private absolute paths.
- [ ] README does not mention unimplemented TUI/browser/component-mix behavior as available.
